# Isolate local Backends behind platform-aware runners

The orchestrator will remain independent of model runtime dependencies and execute each local Backend through an isolated runner process or managed local server. A Backend may provide a native Windows runner, a WSL2 runner, or both; native Windows is first-class when it passes the shared conformance tests, and WSL2 is a supported fallback rather than a global requirement. Runner termination provides a reliable VRAM-release boundary and lets llama.cpp, Transformers, Diffusers, speech, and alignment stacks use compatible environments without contaminating the orchestrator. This adds launcher and path-mapping complexity, but avoids a fragile all-model Python environment and preserves the ability to mix local and cloud Backends in one Run.
